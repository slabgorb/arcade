# Narrative

## Problem Statement
**Problem:** A reference document our team relies on when building Joust — the "process block" spec, essentially the blueprint for how every moving character (bird, egg, enemy) is represented in memory — had the wrong math baked into two of its foundational statements. It said each entity's data record was 51 bytes; it's actually 56. **Why it matters:** This isn't cosmetic — the upcoming flight and ground-movement feature (story jt1-5) is about to build its collision and movement logic directly on top of this spec. If the wrong number shipped into that feature, characters' positions on screen could end up subtly miscalculated, and the bug would be very hard to trace back to its root cause because it would look like a movement bug, not a documentation bug. Catching it now, before jt1-5 starts, is far cheaper than finding it after the movement code is built and tested.

## What Changed
Think of the process-block spec like a floor plan for an apartment that a contractor is about to build from. The original 1982 arcade engineers left a note on their own blueprint that said "8 square feet of hallway" — but if you actually measure every room on the blueprint itself, the hallway is 7 square feet, and two entire rooms (the X and Y position fields) were left off the total square footage entirely. Our dossier had been quoting the note ("8 square feet") instead of measuring the rooms.

We corrected the dossier to state the true numbers — 7 bytes of overhead, 56 bytes total per entity — and we kept the old, incorrect note visible in the document, clearly labeled as outdated, so future readers can see exactly why the mistake happened and won't rediscover it themselves. We also re-pointed every supporting citation so it backs up the corrected numbers with exact source references, and added ten new supporting references covering fields that weren't documented before. Nothing in the actual game changed — this is purely a correction to internal reference material the team uses to build features correctly.

## Why This Approach
The team could have simply deleted the wrong note and quietly typed in the right numbers. Instead, we deliberately kept the original (wrong) vendor note visible in the document, labeled as stale, right next to the correction. The reasoning: this exact error survived unnoticed through four prior pieces of work, because our verification tooling only checks that a quoted reference is copied word-for-word accurately — it doesn't check whether the surrounding explanation drawn from that quote is actually correct. Leaving the wrong note visible, with an explanation of why it's wrong, means the next person who reads this document sees the trap and doesn't fall into it again. We also had three different people independently re-derive the math from scratch (rather than trust each other's numbers or the original summary) and all three arrived at the same corrected total — 56 bytes — which is about as much confidence as this kind of correction can get.
