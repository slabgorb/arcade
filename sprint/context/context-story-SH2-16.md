## Problem
First audio subpath and a BROWSER subpath (touches AudioContext), fenced by the purity guard like glow/view/esc-overlay. Design: docs/superpowers/specs/2026-07-11-shared-audio-extraction-design.md. Four games hand-write a near-identical shell-side WebAudio SFX engine (tempest/star-wars/asteroids/battlezone); extract the shared VERB into createAudioEngine(manifest) -> { resume, play, startLoop, stopLoop, ready } with getAudioContextCtor (webkitAudioContext prefix), lazy-context-on-gesture, silent-degrade at EVERY failure path (no WebAudio, blocked autoplay, failed fetch, undecodable sample all leave the game silent, never throw), a master GainNode, buffer loading/decoding, and POKEY-style channel/voice-stealing (stopChannel/startSource/onended). Generic over the game's SoundName so play() stays typed at the consumer: createAudioEngine<N extends string>(manifest: { baseUrl, masterGain?, sounds: Record<N,string>, channels: Record<N,string> }) -> AudioEngine<N>. Per-cabinet NUMBERS (the SOUNDS name->file manifest, CHANNELS map, baseUrl, masterGain) pass in as config and stay in the game. Key the internal buffer store by FILENAME so multiple names -> one .wav decode once (proves asteroids' N:1 case ahead of SH2-17). Re-point tempest: delete its local src/shell/audio.ts engine body, keep its manifest + audio-dispatch.ts (event->sound wiring stays game-side), and prove the module against a real consumer — tempest's tsc build typechecks the generic. Speech is OUT OF SCOPE (star-wars-only, deferred to SH2-17). Extend the purity guard so audio is a recognised browser subpath (exempt) while the pure core stays DOM-free. arcade-shared tests are untyped (vitest strips types) — pin the mechanism at runtime via a fake AudioContext, not compile-only annotations.

## Acceptance Criteria
- @arcade/shared/audio exports AudioEngine<N> and createAudioEngine<N extends string>(manifest) -> AudioEngine<N>, where manifest = { baseUrl, masterGain?, sounds: Record<N,string>, channels: Record<N,string> }; the ./audio subpath is added to package.json exports (built ESM + .d.ts) and the prepare build, and the package version is bumped.
- A fake-AudioContext test pins the engine mechanics: the context is created lazily (only on resume(), never before); play/startLoop/stopLoop are silent no-ops before resume() and when WebAudio is absent; a retrigger on a channel STEALS (stops the prior source); startLoop sets source.loop and stopLoop stops it; and every failure path (missing buffer, decode reject, stop() on an ended source) degrades silently without throwing.
- Buffers are keyed by filename so a manifest mapping multiple logical names to one file fetches/decodes that file once; a test asserts the single decode and correct name->file->buffer resolution (the asteroids case).
- The purity guard recognises audio as a browser subpath (exempt) and still fails the pure core (math3d/rng/highscore/loop/font) on any DOM reference.
- tempest imports @arcade/shared/audio at a pinned git-URL ref; its local src/shell/audio.ts engine body is deleted (its SOUNDS/CHANNELS manifest + audio-dispatch.ts retained, audio.ts now constructs/re-exports from the shared module); tempest vitest + vite build are green and a manual run confirms SFX + the sustained loops (pulsar hum, warp/zoom) are audibly unchanged.
