## Problem
rb4-2 retracts the findings doc's 'built-in distance LOD' claim (which the doc itself only ever tagged '[LOD reading inferred]'). This story retires the BEHAVIOUR that grew out of it. The ROM does not test distance anywhere in the picture path. DRNPIC (RBARON.MAC:4961-4963, citable copy) reads 'LDA PLSTAT+6 ;PLANE ROTATED' then 'AND I,10' — and in a .RADIX 16 region that 10 is 0x10, bit D4, set/cleared at RBARON.MAC:2652 ';D4=0 (PLANE FACING AWAY)'. Bit clear takes the 20-branch: .DRPNT, the 29-point drone point-list, drawn with the DB.MAR front list. Bit set: scale-down plus DB.MAP with back faces. So the far/near model is chosen on ORIENTATION — is the plane turned away from you — not on how far away it is. src/core/biplane.ts instead carries LOD_DISTANCE, a depth threshold: first 1500 (invented outright), then re-derived by rb4-1's Reviewer as a projected-wingspan fraction — a better-argued version of a test the ROM never makes. src/core/enemy.ts:83 now cites it as precedent for another tunable. NOTE the trap rb4-1 already hit: biplane.ts's behavioural tests passed IDENTICALLY at 1500 and at 1584 — they only ever asked that the switch land somewhere inside the flight band. A test that stages the plane at a convenient depth CANNOT see this defect; pin the MODEL CHOICE against the orientation state.

## Acceptance Criteria
- The plane model is selected on a 'rotated / facing away' orientation state mirroring PLSTAT+6 bit 0x10 — not on camera-space depth. A plane at ANY distance draws the 29-point .DRPNT model when facing away, and the full model when not.
- LOD_DISTANCE is gone from src/core/biplane.ts, and enemy.ts:83 no longer cites it as precedent for a tunable.
- A test proves the ROM's rule bites where a distance rule cannot: same depth, two orientations, two different models. A depth-threshold implementation must FAIL it — mutate the old rule back in and confirm it turns red.
- The near model draws the back-face list (DB.MAP) and the far model the front list (DB.MAR), per DRNPIC's two branches.
