## Title
Spike + ADR: how the lobby reads each game's high score across origins (no backend)

## Problem
Gate story for the epic — decide, in writing, how the lobby can read each game's best score when the lobby and the games are separate origins in production (arcade.slabgorb.com vs <game>.slabgorb.com) and the cabinet has no backend. Spike the candidates far enough to know which ones actually survive a modern browser, then land an ADR: (a) a cookie on the shared parent domain .slabgorb.com, readable by every subdomain but size-capped and sent on every request; (b) a hidden-iframe + postMessage bridge to one shared storage origin — verify against third-party storage partitioning in Chrome and Safari ITP before betting on it; (c) collapse the cabinet onto a single origin using the path prefixes the vite configs already carry (arcade.slabgorb.com/tempest/ ...), which makes the whole problem disappear at the cost of the per-game R2 bucket/subdomain layout in docs/ops/hosting.md; (d) a Cloudflare KV/Worker score service, which breaks the standing no-backend constraint and therefore needs an explicit exception. No code — the deliverable is the decision and its blast radius, because whatever wins is implemented across @arcade/shared/highscore, the lobby, and potentially all five games.

## Acceptance Criteria

- Each candidate is spiked far enough to be falsifiable: the postMessage/iframe option is tested against (or cited against) third-party storage partitioning in current Chrome and Safari, and the parent-domain-cookie option is checked against a realistic five-game score payload.
- The ADR names the blast radius explicitly: which repos change (@arcade/shared/highscore, lobby, and which games), whether docs/ops/hosting.md's per-subdomain R2 layout is affected, and whether the no-backend constraint is being kept or knowingly excepted.
