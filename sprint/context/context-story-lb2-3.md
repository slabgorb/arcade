# Story lb2-3 Context

## Title
Lobby tiles re-read high scores on return from a game (kill the frozen-at-load read)

## Metadata
- **Story ID:** lb2-3
- **Type:** bug
- **Points:** 2
- **Priority:** p2
- **Workflow:** tdd
- **Repo:** lobby
- **Epic:** Lobby follow-ups — cross-origin high scores, plus front-end/design work

## Problem
The 'frozen' half of the original report, now downstream of ADR-0004. lobby/src/main.ts calls getTopScore(game.id) once while building each tile, so the value is fixed for the life of the page: play a game, beat your best, come back — the tile still shows the old number, and on a BFCache restore the page is not even rebuilt. Make the tiles re-read when the player returns: a pageshow handler covers BFCache back-navigation. Post-ADR-0004 the source is a cookie, so cross-tab 'storage' events do NOT apply (there is no storage event for cookies) — do not build around one. Re-read through a single entry point that re-uses the tile's existing render path. Small, self-contained, and shippable independently of lb2-2.

## Technical Approach

_Constraints and landmarks only — TEA and Dev own the design._

### Landmarks (verified in this checkout)
- `lobby/src/main.ts:8` — imports `getTopScore` from `./shell/storage`.
- `lobby/src/main.ts:17` — `renderTiles(games, GAMES, getTopScore)`. This is the
  one build-time read. Tiles are frozen from here on; nothing re-reads.
- `lobby/src/shell/storage.ts:31` — `getTopScore(gameId: string): number | null`.
  It returns `null` when the score is unreadable. AC3 hinges on `null` staying
  distinct from `0` — an unreadable tile must show NO SCORE, never a `0`.

### Binding constraints
- **The source is a cookie**, per [ADR-0004](../../docs/adr/0004-cross-origin-high-scores.md)
  (Accepted, from lb2-1; lb2-2 shipped it to production). Cookies emit **no
  `storage` event** — there is nothing to listen to cross-tab. Do not build
  around one, and do not "fix" this by reverting the read to `localStorage`.
- **`pageshow` is the trigger**, and it must handle `persisted === true`
  (BFCache back-navigation), where the page is *not* rebuilt and a load-time
  read never re-runs. A cold-load-only fix does not satisfy AC1.
- **One re-read entry point**, re-using the tile's existing render path. Do not
  rebuild the grid and do not duplicate the score formatting that the render
  path already owns (AC2).

### Known trap
`null` and `0` are both falsy. A refresh path that coalesces (`score || 0`, or a
truthiness check) will regress AC3 by flickering an unreadable tile to `0`. The
distinction must survive the re-read.

## Scope
- **In scope:** re-reading high scores on return to the lobby (incl. BFCache) via
  a single entry point over the existing tile render path, plus tests that drive
  that path with a simulated `pageshow`.
- **Out of scope:** the cookie/storage contract itself (settled by ADR-0004 and
  shipped in lb2-2), cross-tab live updates, and any tile visual redesign.

## Acceptance Criteria
- Returning to the lobby from a game shows the newly-set high score without a manual reload — verified across a BFCache back-navigation (pageshow with persisted=true), not just a cold load.
- Refresh goes through one re-read entry point re-using the tile's existing render path — no duplicated score formatting, no rebuilding the whole grid.
- A tile whose score is still unreadable stays at NO SCORE across refreshes and never flickers to a stale value or a 0.
- Tests drive the refresh path directly (simulated pageshow), asserting the tile text changes from the old score to the new one.
