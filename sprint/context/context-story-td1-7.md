## Problem
Found by the td1-1 Reviewer (2026-07-20). battlezone/tests/scaffold.test.ts:53, red-baron/tests/scaffold.test.ts:70 and centipede/tests/scaffold.test.ts:69 each assert 'strictPort: true' — the guarantee td1-1 just PROVED INSUFFICIENT — while NOTHING asserts host: '127.0.0.1', the guarantee that actually works.

WHY THIS MATTERS: the suites currently guard the broken invariant and leave the working one unguarded. A future edit could silently delete the host pin and regress the exact bug td1-1 fixed, with a fully green suite. That is the same failure shape td1-4 found in the audit tooling (a check that cannot fail is worse than no check) and the same shape jt1 flagged repeatedly.

THE PROOF td1-1 established, at the socket layer, both directions:
  held 127.0.0.1:19911 -> binding vite's default host 'localhost' SUCCEEDED on ::1  (port shared, NO error)
  held 127.0.0.1:19912 -> binding '127.0.0.1' gave EADDRINUSE                       (fails loudly)
So the pin is the load-bearing line; strictPort alone is not.

SCOPE: add an assertion for host: '127.0.0.1' on BOTH the server and preview blocks to the three repos that already have a scaffold.test.ts. Correctly deferred from td1-1 (a 2-point mechanical config port) rather than scope-creeping it.

WORTH CONSIDERING: lobby, tempest, star-wars and asteroids have no scaffold.test.ts at all, so they carry no guard of either kind. Decide explicitly whether to extend the pattern to them or leave them unguarded — and record the decision either way rather than leaving it implicit.

A BEHAVIOURAL test (bind 127.0.0.1, assert the dev server fails loudly) would be stronger than a regex check on the config text, but note joust's own behavioural strictPort test occupies IPv4 only and was flaky before the host pin — copy that fix if the test is ported.
