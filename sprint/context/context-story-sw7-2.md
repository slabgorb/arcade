## Technical Approach
One root cause, one root fix — this is a 2-pt bug, not a redesign.

- **The defect.** The ROM's music/speech wave gates read the **0-based** hardware
  counter `GM.WAV`. `CMPA #4-1` gates "wave 4+", i.e. 0-based index `>= 3`, which in
  our **1-based** wave numbering is the set `{4, 6, 8, ...}`. Our code instead tests a
  1-based wave with `wave % 2`, producing a set that is **disjoint / inverted** from the
  ROM. Symptoms: the Imperial March music cue, the "Great shot, kid!" speech line, and
  **both** trench voice-line sets fire on the wrong or inverted waves. Per **U-008's
  correction**, the even-wave trench cues are specifically **INVERTED**, not merely
  mismatched.
- **The fix shape (for TEA/Dev — SM does not prescribe code).** Introduce **one**
  0-based wave accessor (or a `-1` shim) in `src/core`, and rewire the three gate sites
  — Imperial March music, "Great shot, kid!" speech, both trench voice-line sets — to
  read wave parity through it. Replace the ad-hoc `wave % 2` parity tests. Make the shim
  the single mutation point (flip it → a test must go red).
- **Evidence & citations.** Findings **U-005, U-006, U-007, U-008** all live in
  `docs/audit/findings/pair-audio.json`. Fixing a fix-class finding = set
  `remediated_by: sw7-2` on it and keep `npm test -- citations` green. Do **not** mark a
  CONFIRMED-faithful (no-code-change) finding as remediated.
- **Prerequisite (already landed).** sw7-1 (timebase reconcile to 20.508 Hz) is merged to
  `origin/develop` (#91). Branch off current `origin/develop`. No timebase work here.
