## Problem
cp1-6 review carry-forwards R1/R4/R5. R1 (src/main.ts:61-71): input is sampled ONCE per rAF frame — on >60 Hz displays the 0-step frames drain-and-DROP accumulated movementX (about 50 percent sensitivity loss at 120 Hz), and a catch-up burst after a tab-unhide re-applies the SAME delta to up to ~15 sub-steps (gun teleport). Fix: deltas accumulate in the adapter until a sim step consumes them; each step drains exactly once (level inputs sampled per step; deltas never reused across steps). R4: canvas.requestPointerLock() (main.ts:36) returns a Promise whose rejection (re-lock cooldown after Escape) is unhandled. R5: input.ts's blur-handler comment claims to cover pointer-lock drops, but no pointerlockchange listener is wired — an Escape-exit that keeps window focus never clears held state. The pointer-lock path has NEVER been live-verified (headless browsers reject it — cp1-6 Dev finding): this story ends with a HUMAN smoke test by the user in a real browser tab.
