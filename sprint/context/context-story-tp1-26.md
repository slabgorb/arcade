## Problem
Found by Obi-Wan during the tp1-5 review and left homeless; filed on Jedi's instruction (2026-07-14). tp1-5 landed W-026 (the pulse is ONE global counter, PULSON stepped by PULTIM, lit for 7 frames of 40) and W-027 (a pulse only kills from inside the potency zone, PULPOT). Both are correct AT WAVE 1 AND WRONG LATER, because both constants are wave-parameterised in the ROM's WTABLE and we hardcoded the first row. WPULTIM (ALWELG.MAC:610-613): `.BYTE T1,1,48.,4 / .BYTE T1,49.,64.,6 / .BYTE T1,65.,99.,8` — PULTIM is 4 for waves 1-48, then 6, then 8. `PULSE_STEP = 4` (src/core/rules.ts) is the first row only. Since PULTIM is the counter's STEP, changing it changes the period AND the duty cycle AND the reachable residue class — the whole '7 frames of 40' derivation is a wave-1 result, not a universal one. WPULPOT (606-609): `.BYTE T1,1,32.,0A0 / .BYTE T1,33.,64.,0A0 / .BYTE T1,65.,99.,0C0` — PULPOT is $A0 through wave 64 and $C0 from 65. `PULSAR_NEAR_FAR_DEPTH` hardcodes $A0. AND IT IS REACHABLE: nothing caps the level (sim.ts increments s.level on every clear; MAX_SELECT_LEVEL=16 only bounds the STARTING wave). A player who survives past wave 48 gets a pulse on the wrong beat and, past 64, a potency zone in the wrong place. SECOND, SMALLER PROBLEM, same family: `PULSAR_NEAR_FAR_DEPTH` currently serves TWO distinct ROM constants — JPULMO's climb-speed threshold and PULPOT (the W-027 kill gate). They coincide at $A0 today and DIVERGE at wave 65, so a future correction to one will silently move the other.

## Acceptance Criteria
- PULTIM is read per wave from WPULTIM (4 / 6 / 8 at waves 1-48 / 49-64 / 65-99), cited. The wave-1 behaviour tp1-5 pinned (period 40, lit for 7, seeded at -1 by INEWLI) must NOT regress.
- PULPOT is read per wave from WPULPOT ($A0 for 1-64, $C0 for 65-99), cited, and is given its OWN name — it must stop sharing an identifier with JPULMO's climb-speed threshold, which is a different ROM constant that merely happens to equal it below wave 65.
- A test derives the pulse period and duty cycle OUT OF THE RUNNING SIM (never by re-deriving the arithmetic in the test — that is how the audit and its own refuter both got 9 instead of 7) at a wave in each PULTIM band, and asserts they differ across the bands.
- A test pins the kill gate at a wave on each side of 65, so the $A0 -> $C0 step is real and not decorative.
- npm test -- citations stays green.
- READ THIS BEFORE YOU PORT THE TABLES — tp1-25 already fell into the hole waiting for you. Both WPULTIM (ALWELG.MAC:610-613) and WPULPOT (606-609) END AT WAVE 99, and s.level is NOT capped (sim.ts increments it on every clear; MAX_SELECT_LEVEL bounds only the SELECT screen). A naive table walk returns its end-of-table 0 at wave 100 — and 0 is far nastier here than it was for the fuseball: PULTIM is the pulse counter's STEP, so a step of 0 means the counter never advances and THE PULSE FREEZES FOREVER; PULPOT of 0 puts the kill zone at depth 0. The ROM never reaches the end of its own table because CONTOUR (415-423) folds it first: for CURWAV >= 98 it substitutes a RANDOM wave, `LDA RANDO2 / AND I,1F / ORA I,40` then INC, giving 65..96 — a band that lies wholly inside each table's last record, so the draw is UNOBSERVABLE and the answer is deterministic. Every table lookup must go through that fold.
- Do NOT re-derive the fold per table. tp1-25 shipped it inline in wfuschForLevel (src/core/rules.ts) because it was the only bounded table at the time; this story makes it three. EXTRACT it once — a `contourWave(level)` helper carrying the ALWELG.MAC:415-423 citation — and route wfuschForLevel, WPULTIM and WPULPOT through it. The bug is in the SHAPE of the port, not in any one table, and the next table will inherit it.
- A test drives each of the three lookups at a wave ABOVE the last record (100, 150) and asserts the ROM's answer, not 0. wfuschForLevel's existing deep-wave tests (tests/core/tp1-25.fuseball-chase.test.ts) must still pass UNCHANGED after the extraction.
