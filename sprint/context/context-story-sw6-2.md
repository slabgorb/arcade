# Story sw6-2 Context

## Title
A music cue that arrives before its buffer decodes is lost forever — the first-ever visitor hears no space theme, because @arcade/shared/audio's startSource silently no-ops on an unloaded buffer

## Metadata
- **Story ID:** sw6-2
- **Type:** bug
- **Points:** 3
- **Priority:** p1
- **Workflow:** tdd
- **Repo:** arcade-shared, star-wars
- **Epic:** Star Wars — ROM audio fidelity (the music that has never played)

## Problem
sw6-1 baked the four loops and put them live on R2 — and a first-time visitor STILL hears no space theme. Found by Dev during sw6-1 by actually playing the game, not by a test. The music buffers (~5 MB) only begin fetching when `resume()` calls `load()` (arcade-shared/src/audio.ts:80,116), and `resume()` is wired to the first user gesture (star-wars/src/main.ts:57). On that very same keypress the core fires its run-start MusicEvent, so `startLoop('space')` (main.ts:223) reaches `startSource` (audio.ts:138) BEFORE the buffer exists, hits `if (!buffer) return  // silent no-op` (audio.ts:141), and the entire space phase plays in silence. The second run has music, because by then the buffers have decoded — which is exactly why this survived sw6-1's browser check. This is the SAME silent-degrade contract that hid this epic's 404s for a full epic: the assets are now genuinely correct, and the engine is now hiding a load race instead of a missing file. The engine is shared, so every consumer whose first music cue rides the unlock gesture has this bug, not just star-wars.

## Acceptance Criteria
- REPRODUCE IT FIRST, IN A TEST — a failing test pins the actual defect: `startLoop(name)` issued BEFORE that name's buffer has decoded currently plays nothing, ever. Note arcade-shared's tests are untyped and run in a node env (no root tsconfig; vitest strips types), so pin the contract at RUNTIME, never with a compile-only annotation.
- A LOOP REQUESTED EARLY IS HONOURED LATE — `startSource` remembers the requested loop and starts it when that buffer finishes decoding (audio.ts:89, where the decode resolves). Scope this to the LOOP path only: a one-shot `play()` that arrives early must still be DROPPED. A laser that fires before its sample decodes should not fire half a second late, out of sync with the shot that caused it — music is sustained and positional in time, a sound effect is not.
- LAST REQUEST WINS, AND THE SINGLE-MUSIC-CHANNEL INVARIANT HOLDS — if two loops are cued before either decodes (a fast phase edge on a cold load), only the most recent one may sound. sw3-5's invariant is one looping source at a time on the `music` channel; a pending-start queue must not resurrect a stale track behind the current one.
- A LOOP CANCELLED BEFORE IT DECODES MUST NEVER START — `stopLoop()`, or a phase edge that swaps the track, between the request and the decode cancels the pending start. Otherwise the fix introduces a new bug: music that fades up seconds after the phase that wanted it has ended.
- `ready()` IS NOT THE FIX, AND IS ITSELF MISLEADING — `ready()` returns `buffers.size > 0` (audio.ts:175), i.e. TRUE as soon as ANY ONE sound has decoded. The tiny SFX always win the race against the ~5 MB music, so gating START on `ready()` would NOT fix this and would look like it had. Either make `ready()` mean what its name says, or leave it and record explicitly in the session why it cannot be the gate.
- A MISSING ASSET MUST STILL BE DISTINGUISHABLE FROM A SLOW ONE — the decode path already swallows a failed fetch (audio.ts:93, `one missing sound is non-fatal`). Do not turn the pending-start into a queue that waits forever and thereby hides a 404 the way this epic's original bug was hidden. A request for a sound that FAILED to load must not sit pending indefinitely with no trace.
- THE FIX IS SHARED, SO THE FLEET INHERITS IT — the change lands in `arcade-shared` and ships as a tagged version; star-wars re-pins to that tag in the same PR-set (git-URL dependency, so force the install: `npm install "@arcade/shared@github:...#ref"`). Any other game whose first music cue rides the unlock gesture gets the fix for free on its next re-pin; say in the session which ones were checked.
- THE ACCEPTANCE TEST IS A COLD LOAD IN A REAL BROWSER, NOT A GREEN VITEST — hard-refresh with an EMPTY CACHE (a warm cache is precisely what hides this), press START on the very first keypress, and hear the space theme from the first note of the first run. No unit test can prove this; verify it in the browser and say so in the session. Port trap: a pinned arcade port may be served by a SIBLING CHECKOUT (a-2/a-3), so confirm the server's cwd with `lsof` before trusting what you hear.
