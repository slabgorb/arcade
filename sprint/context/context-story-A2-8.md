## Technical Approach

This is a ROM-faithful implementation of the Asteroids shrapnel system (distinct from ship debris, which A2-5 already ported).

**Key Findings (recovered from 1979 ROM by Architect):**

1. **Two separate debris systems:**
   - Ship explosion: drifting line fragments (`DoShipExplsn` $7465) — already ported as `src/core/shipDebris.ts` (A2-5)
   - Asteroid/saucer explosion: **SHRAPNEL** (`DrawObjectExplode` $7349 → `ShrapPatPtrTbl` $50F8) — **this story's scope**

2. **Shrapnel mechanism:**
   - ~10-11 STATIONARY bright dots scattered around impact point
   - Each dot = dark reposition + zero-length `SVEC …b=7` point
   - Lit at intensity b=7 (dimmer than ship fragments' b=12) — "dim/subtle" is ROM-consistent
   - Does NOT drift (skips `UpdateObjPos` in ROM)
   - Expansion via SCALE: timer counts up $A0→positive, ~20 frames ≈ 0.3-0.4s (short-lived)
   - Scale formula: `(timer & $F0) + $10` → 5 growth steps
   - Pattern cycling: `index = (timer & $0C) >> 1` → 1 of 4 patterns/frame

3. **Spawn trigger:**
   - Hang off the SAME `ExplosionEvent` (already fired at `asteroids/src/core/sim.ts:341-344`)
   - Do NOT add a second detector

4. **Implementation guidance:**
   - Reuse the STRUCTURE of `shipDebris.ts` (transient-piece state array + `update*(dt)` aging + render fade-by-life + `spawn*()` fn)
   - NOT its drift physics — shrapnel is stationary + expanding + point-based
   - Keep RNG-FREE (ROM shrapnel deterministic; RNG perturbation risks wave-spawn determinism like A2-6 did)
   - Fidelity scope: full 4-pattern cycle vs single-pattern scale-expansion; "dim" brightness is the aesthetic knob

5. **ROM reference (raw asm):**

   Then search for "Shrapnel Patterns Vector Data" section for exact coordinates.
