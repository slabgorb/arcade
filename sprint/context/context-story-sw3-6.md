## Problem
The sw2-6 disassembly fidelity audit flagged two missing HUD-scoring behaviors
in the ported star-wars sim (`docs/sw2-6-disassembly-fidelity-audit.md` lines
54-61, 208-212):
1. **No extra lives.** The 1983 cabinet awards a bonus life at 400,000 and again
   at 800,000. Today `lives` only ever decreases (`grep 400000|800000|extraLife`
   = 0 hits).
2. **No flashing bonus/extra-life HUD row.** `byte_4B2C` is the "score changed,
   redraw HUD" flag / a flashing yellow-amber counter drawn directly under the
   6-digit score. The findings doc calls it out explicitly: "NOT modelled in
   core/state.ts (no bonus/extra-life field) — a genuine gap for a future HUD
   task." This story is that task.

## Technical Approach
_Pointers for TEA/Dev — the ROM is authoritative; TEA owns the failing tests._

**Authoritative sources (in the star-wars subrepo):**
- `docs/star-wars-1983-source-findings.md` ~405-490: the score adder `loc_9810`
  (`byte_4B2C` set on every score change, line ~415); the `byte_9865` "Death Star
  destroyed incrementing score value" table = 5,000 / 200,000 / 400,000 / 600,000
  / 800,000 (the 200k & 600k are **dead entries**); extra-life text
  `a40000`/`a80000` = **400,000 / 800,000**.
- `docs/star-wars-1983-source-findings.md` ~479-483 (`sub_761D` "Display score"
  draws the score panel, then the flashing `byte_4B2C` counter under it, then the
  wave digit) and ~665-683 (the HUD-gap note; bonus row is flashing YELLOW/amber).

**Load-bearing trap — do NOT ×10.** The findings doc resolves a cross-note
conflict: thresholds are exactly **400,000 / 800,000**, not 4M/8M or 250k/500k.
Two independent readings (on-screen "SCORING" text + the BCD arithmetic) agree.
See the ⚠︎ conflict note (~451-463) and epic memory. Each threshold awards +1
life exactly once (a single delta that vaults past both grants both).

**Seams (do not implement — for orientation only):**
- `src/core/state.ts` — `GameState` has `score`/`lives` (~423-424), `STARTING_LIVES`
  (~530). Add the bonus-row / `byte_4B2C`-analog field here.
- `src/core/sim.ts` — score is incremented at ~237 (TIE), ~265 (fireball), ~453
  (turret). Extra-life awarding + bonus-row set/decay belong in the `stepGame` tick.
- `src/core/hud.ts` + `src/shell/render.ts` — HUD surfaces; the shell renders the
  flashing bonus row from core state only.

**Hard boundary (star-wars/CLAUDE.md):** `src/core` is a pure deterministic sim —
no DOM/window/canvas, no `Date.now`/`performance.now`/`Math.random`/`rAF`. All
extra-life + bonus-row logic (including flash/decay) lives in core and updates
from ticks; the shell only reads state to render. The `byte_4B2C` −8/refresh
intensity fade is a cosmetic detail — decay behavior is required, but do not
over-pin the exact per-refresh rate.

## Acceptance Criteria
- Crossing a cumulative score of 400,000 awards exactly one extra life the first time it is reached; it is never re-awarded on later score gains.
- Crossing 800,000 awards exactly one additional extra life, once. The 400k and 800k awards are independent and each fires exactly once (a single score delta that vaults past both grants both lives). Thresholds are exactly 400,000 / 800,000 per byte_9865 — NOT multiplied by 10.
- A bonus-row value (the byte_4B2C analog) is added to GameState: on a score-changing event it records the awarded amount, then decays/flashes out toward zero over subsequent ticks. Deterministic and tick-driven (no DOM/time/random).
- The HUD renders a flashing yellow/amber bonus/extra-life row directly beneath the 6-digit score value, driven by the new state field; when the value has decayed to zero the row is absent.
- All extra-life and bonus-row logic lives in src/core (updated from ticks via stepGame); src/shell only reads state for render. The core purity boundary is preserved — no DOM, Date, performance.now, or Math.random added.
- Existing scoring values (TIE 1,000 / fireball 33 / turret 100 / etc.) are unchanged and all current tests stay green.
