## Title
Trench pilotable viewpoint (sub_703B ±511 lateral / −257..−3583 vertical band) — makes catwalks dodgeable instead of a guaranteed shield loss every run

## Problem
The trench run currently has no pilotable viewpoint. The player cannot move
laterally/vertically to dodge the catwalks, so every trench run costs a
**guaranteed shield loss**, and — because nothing varies frame-to-frame — every
run is byte-identical. This divergence was surfaced by the sw2-6 disassembly
fidelity audit.

## Technical Approach
_Approach hints to be refined by TEA/Dev — the ROM behavior below is the source of truth._

- **Source of truth:** `star-wars/docs/sw2-6-disassembly-fidelity-audit.md`
  (the audit that spawned epic sw3). The ROM routine is `sub_703B`.
- **ROM behavior:** `sub_703B` updates the player's trench-frame position each
  tick within a clamped band — **lateral ±511**, **vertical −257..−3583** — with
  no wrap/overshoot at the edges.
- **Placement:** the deterministic core sim (`src/core`) owns the position
  update; `src/shell` only reads it for render/input. Follow the existing
  star-wars core/shell split (see the sw2 epic sessions for the trench-phase
  entry points).
- Related prior work: sw2 wired the trench phase, TIE/fireball mechanics, and
  the aim/lock math — this story adds the missing pilotable band on top.

## Acceptance Criteria
- Player viewpoint in the trench phase is pilotable — lateral and vertical position update from input each tick within the ROM clamp band (lateral ±511, vertical −257..−3583).
- Position is clamped to the band edges (no wrap, no overshoot).
- Catwalks/obstacles become dodgeable: a run where the player steers clear does NOT incur the previously-guaranteed shield loss.
- Deterministic core sim (src/core) owns the position update; src/shell only reads it for render/input. Follows existing star-wars core/shell split.
