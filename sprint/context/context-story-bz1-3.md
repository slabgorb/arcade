## Problem
bz1-2 quarried the ROM's data into pure core modules (`math3d.ts`, `models.ts`,
`obstacles.ts`) but nothing renders yet. This story builds the render
foundation: a camera derived from the tank pose, a single perspective pipeline
through the ported Math Box, the ROM's camera-relative culling window, the
horizon skyline (mountains / volcano / moon), and the wireframe 21-obstacle
field — projected in pure core, stroked by the shell.

## Technical Approach
_Refined by TEA (Furiosa) during RED — contract decisions pinned by the failing suites:_

- **`src/core/camera.ts`** — `TankPose { x, z, heading }` → view matrix via the
  ported `viewMatrix()`. ROM conventions (findings §4, mathbox.html verbatim):
  heading 0 faces **+Z**, increasing heading rotates counter-clockwise toward
  **+X**, and **+X renders screen-LEFT**. The eye sits at world **y = 0**: the ROM
  bakes eye height into vertex data (every ground-sitting model's base plane is
  y = −640), so `GROUND_Y = −640` and no eye-height translation exists.
  Exports ROM constants `NEAR_CULL = 1023` ($03ff), `FAR_CULL = 31487` ($7aff),
  `H_FOV = π/4` (the confirmed 45° visibility cone).
- **`src/core/scene.ts`** — `sceneProjection(aspect)` (45° horizontal FOV,
  aspect-corrected); `OBSTACLE_MODEL` (ObstacleType → Model3D, exhaustive);
  `visibleObstacles(pose)` (ROM distance-window cull, camera-relative);
  `projectModel(model, placement, pose, aspect)` → NDC `SceneSegment[]` with a
  behind-camera guard; `obstacleSegments(pose, aspect)` for the full field.
- **`src/core/horizon.ts`** — AUTHORED panorama (the quarry has no horizon
  picture data): `PANORAMA` polylines in angular space (azimuth/elevation) named
  `mountains` / `volcano` / `moon`; `panoramaToNdc(azimuth, elevation, heading,
  aspect)` point mapper (null outside view); `skylineSegments(heading, aspect)`
  including a full-width horizon line at NDC y = 0 (eye-level, flat plain).
  `BACKDROP_FOV` may be `H_FOV` or `2·H_FOV` (the epic's "90° FOV horizon" claim
  is quarry-unconfirmed — findings §10; bz1-12 trues this up).
- **Shell** — `render.ts` strokes core-projected segments as glowing green
  vectors on black; verified by eyeball in dev (:5276), per repo convention.

## Acceptance Criteria
1. **Camera from pose (core-tested):** `tankView({x,z,heading})` maps a point
   dead ahead at eye level to eye-space (0,0,−d); heading 0 faces +Z; world +X
   appears screen-left; turning toward +X moves centered objects right;
   deterministic.
2. **ROM ground plane (core-tested):** `GROUND_Y === −640`; ground points
   project below the horizon and converge to it with distance.
3. **45° projection (core-tested):** eye-space bearing ±22.5° lands at NDC
   x = ±1 for any aspect ratio.
4. **ROM culling window (core-tested):** objects strictly outside
   camera-relative [1023, 31487] produce no segments; behind-camera objects
   produce no segments; `visibleObstacles` honors the window against the real
   21-entry table.
5. **21-obstacle field (core-tested):** `OBSTACLE_MODEL` maps all four
   ObstacleTypes to their ROM-decoded models; `obstacleSegments` is non-empty
   from the origin pose, all coordinates finite, byte-deterministic.
6. **Skyline (core-tested, structural):** PANORAMA contains mountains + volcano
   + moon polylines, elevations ≥ 0, mountains leave no azimuth gap ≥ H_FOV;
   skyline output is non-empty at every heading, wraps at 2π, scrolls
   monotonically opposite the turn, includes the full-width horizon line at
   y = 0; deterministic.
7. **Purity (core-tested):** the three new core modules exist and contain no
   DOM/time/randomness tokens and no type-safety escapes (`as any`,
   `@ts-ignore`); they import only from `core/`.
8. **Shell render (eyeball):** `npm run dev` on :5276 shows the green wireframe
   obstacle field + skyline from a fixed viewpoint on black. Not unit-tested
   (repo convention: shell verified by running the game).
