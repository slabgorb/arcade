## Problem
The ported star-wars sim reproduces vertex data, aim/lock, and several scores
faithfully, but diverges from the 1983 cabinet on score **values**. The sw2-6
disassembly fidelity audit surfaced the resolved ROM values. This story bakes
them into the deterministic core sim.

Resolved ROM score values (target → current):
- **TIE fighter kill:** 1,000 (from 100)
- **Darth Vader kill:** 2,000 (new / confirm)
- **Exhaust port (reactor) hit:** 25,000 (from 1,000)
- **Fireball:** 33 (from 50)

## Source of Truth
`star-wars/docs/sw2-6-disassembly-fidelity-audit.md` — the disassembly fidelity
audit that resolved these exact score values against the real cabinet ROM. TEA
and Dev **must** read this audit to confirm each value and locate the current
score constants in `src/core` before writing tests / implementation.

## Technical Approach
- Score values live in the deterministic core (`src/core`). Update the score
  constants there; `src/shell` only reads/displays them.
- Follow the existing star-wars `core`/`shell` split — no scoring logic in shell.
- Update existing scoring tests to the new ROM-resolved values.

## Scope
- In scope: the four score constants above (TIE, Vader, exhaust port, fireball)
  set to their ROM-resolved values, plus the tests that pin them.
- Out of scope: any other scoring behavior, HUD/bonus-row work (that is later
  sw3 stories), and non-scoring changes.

## Acceptance Criteria
- Score constants for TIE (1,000), Darth Vader (2,000), exhaust port (25,000),
  and fireball (33) match the ROM-resolved values from the sw2-6 audit.
- Deterministic core sim (`src/core`) owns the score values; `src/shell` only
  reads/displays them. Existing core/shell split preserved.
- Existing scoring tests updated to the new values; no other scoring behavior
  regressed.
- Build + full test suite green: `cd star-wars && npm run build && npm test`.
- Final AC wording confirmed by TEA against the sw2-6 audit during the RED phase.
