## Problem
Cluster C9. Subsumes SN-003, SN-005, SN-006, SN-007, SN-008, SN-011, SN-012, SN-013, SN-014, SN-016, SN-017. THE CLOCK TRAP: MODSND is called from the NMI handler (RBGRND.MAC:237) at 250 Hz / 4 ms — NOT the 96 ms calc frame. Any envelope timed on the calc frame is 24x wrong. SN-003 is the subtlest finding in the audit and it argues the ROM's OWN COMMENTS are wrong: RBSOUN.MAC:89 and :152 both say NUMBER is '# of changes - 1', but the code renders exactly NUMBER distinct values — proven because under that reading all five sounds' AUDF and AUDC chains run out together (TK 28=28, TP 40=40, BN 288=288, WP 360=360, TH 256=256) and under the documented reading all five mismatch. Our pokey.ts enshrines a `steps = NUMBER + 1` rule in a loud warning comment; the correct rule is `steps = NUMBER - 1`. ANYONE WHO 'FIXES' THIS BACK TOWARD THE ASSEMBLER'S PROSE WILL RE-INTRODUCE THE BUG — record the arithmetic proof in the code. Also: the gun sound fires on EVERY shell including the ENEMY's; the engine hum's timbre is in the source (AUDC=0xA1) and we got it wrong; the explosion fires at the wrong moment and duration; the SPIRAL (shot-down dive whine) is missing.
